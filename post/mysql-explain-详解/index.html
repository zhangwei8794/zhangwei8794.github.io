explain 语句详解

table

无论我们的查询语句多么复杂，里面包含了多少个表，到最后也是对每个表进行单表访问。

所以 EXPLAIN 语句输出的每条记录都对应着某个单表的访问方法。table 列代表表名。

1

3
4

mysql> explain select * from employees;
+----+-------------+-----------+------------+------+---------------+------+---------+------+--------+----------+-------+
| id | select_type | table     | partitions | type | possible_keys | key  | key_len | ref  | rows   | filtered | Extra |
+----+-------------+-----------+------------+------+---------------+------+---------+------+--------+----------+-------+
|  1 | SIMPLE      | employees | NULL       | ALL  | NULL          | NULL | NULL    | NULL | 299157 |   100.00 | NULL  |
+----+-------------+-----------+------------+------+---------------+------+---------+------+--------+----------+-------+

id

查询语句中，每出现一个 SELECT 语句，MySQL 就会为它分配一个唯一的 ID。上面 SQL 虽然用到 inner join 语句，但只有一个 SELECT 查询，所以 ID 值都是 1

1
2
3
4
5
6
7

mysql> explain select * from employees where emp_no in (select emp_no from salaries) or emp_no = 10000;
+----+--------------------+-----------+------------+----------------+---------------+---------+---------+------+--------+----------+-------------+
| id | select_type        | table     | partitions | type           | possible_keys | key     | key_len | ref  | rows   | filtered | Extra       |
+----+--------------------+-----------+------------+----------------+---------------+---------+---------+------+--------+----------+-------------+
|  1 | PRIMARY            | employees | NULL       | ALL            | PRIMARY       | NULL    | NULL    | NULL | 299157 |   100.00 | Using where |
|  2 | DEPENDENT SUBQUERY | salaries  | NULL       | index_subquery | PRIMARY       | PRIMARY | 4       | func |      9 |   100.00 | Using index |
+----+--------------------+-----------+------------+----------------+---------------+---------+---------+------+--------+----------+-------------+

从输出结果可以看到，外层查询有一个独立的 SELECT，子查询有一个独立的 SELECT。所以第二条记录的 id 就是 2

这里需要特别注意，查询优化器可能对涉及子查询的语句进行重写，从而转为连接查询（指的是半连接），比如：

1
2
3
4
5
6
7

mysql> explain select * from employees where emp_no in (select emp_no from salaries);
+----+-------------+-----------+------------+------+---------------+---------+---------+----------------------------+--------+----------+------------------------------------+
| id | select_type | table     | partitions | type | possible_keys | key     | key_len | ref                        | rows   | filtered | Extra                              |
+----+-------------+-----------+------------+------+---------------+---------+---------+----------------------------+--------+----------+------------------------------------+
|  1 | SIMPLE      | employees | NULL       | ALL  | PRIMARY       | NULL    | NULL    | NULL                       | 299157 |   100.00 | NULL                               |
|  1 | SIMPLE      | salaries  | NULL       | ref  | PRIMARY       | PRIMARY | 4       | employees.employees.emp_no |      9 |   100.00 | Using index; FirstMatch(employees) |
+----+-------------+-----------+------------+------+---------------+---------+---------+----------------------------+--------+----------+------------------------------------+

可以看到，虽然查询语句中包含一个子查询，但执行计划中两个表对应的 id 都是 1，这就表明查询优化器把子查询转换成连接查询。

对于包含 UNION 子句的查询来说，每个 SELECT 对应一个 id 也是没错的。

mysql> explain select * from salaries union select * from titles;
+----+--------------+------------+------------+------+---------------+------+---------+------+---------+----------+-----------------+
| id | select_type  | table      | partitions | type | possible_keys | key  | key_len | ref  | rows    | filtered | Extra           |
+----+--------------+------------+------------+------+---------------+------+---------+------+---------+----------+-----------------+
|  1 | PRIMARY      | salaries   | NULL       | ALL  | NULL          | NULL | NULL    | NULL | 2838837 |   100.00 | NULL            |
|  2 | UNION        | titles     | NULL       | ALL  | NULL          | NULL | NULL    | NULL |  442070 |   100.00 | NULL            |
| NULL | UNION RESULT | <union1,2> | NULL       | ALL  | NULL          | NULL | NULL    | NULL |    NULL |     NULL | Using temporary |
+----+--------------+------------+------------+------+---------------+------+---------+------+---------+----------+-----------------+

不过这个执行计划的第 3 条记录是什么意思？为啥 id 是 NULL？而且 table 长的也和别人不一样。

其实 UNION 自举会把多个查询的结果集，合并起来并对结果集中的记录进行去重。UNION 自举为了把 id 为 1 的查询和 id 为 2 的查询的结构合并去重，在内部创建了一个名为<union1,2>的临时表（就是执行计划第三条记录的 table 列的名称），id 为 NULL 表明这个临时表是为了合并两个查询的结果集而创建的。

与 UNION 比起来，UNION ALL 就不需要对最终的结果集去重。它只是单纯地把多个查询结果集中的记录合并成一个并返回给用户，所以也就不需要使用临时表。

1
2
3
4
5
6
7

mysql> explain select * from salaries union all select * from titles;
+----+-------------+----------+------------+------+---------------+------+---------+------+---------+----------+-------+
| id | select_type | table    | partitions | type | possible_keys | key  | key_len | ref  | rows    | filtered | Extra |
+----+-------------+----------+------------+------+---------------+------+---------+------+---------+----------+-------+
|  1 | PRIMARY     | salaries | NULL       | ALL  | NULL          | NULL | NULL    | NULL | 2838837 |   100.00 | NULL  |
|  2 | UNION       | titles   | NULL       | ALL  | NULL          | NULL | NULL    | NULL |  442070 |   100.00 | NULL  |
+----+-------------+----------+------------+------+---------------+------+---------+------+---------+----------+-------+

注意：MySQL 5.6 以及之前的版本，执行 UNION ALL 语句可能也会用到临时表。

UNION 对于包含 UNION 或者 UNION ALL 的大查询来说，除了最左边的查询，其余小查询的 select_type 就是 UNION

partitions

只有分区查询时才会用到，在我们目前遇到的查询语句中，其执行计划的 partitions 列的值都是 NULL。

type

system 当表中只有一条记录并且该表使用的存储引擎（MyISAM、MEMORY）的统计数据都是精确的，那么对该表的访问方法就是 system

const 当我们根据主键或者唯一索引列与常数进行等值匹配时，对单表的访问方法就是 const，比如：

1

3
4
5
6

mysql> explain select * from employees where emp_no=10001;
+----+-------------+-----------+------------+-------+---------------+---------+---------+-------+------+----------+-------+
| id | select_type | table     | partitions | type  | possible_keys | key     | key_len | ref   | rows | filtered | Extra |
+----+-------------+-----------+------------+-------+---------------+---------+---------+-------+------+----------+-------+
|  1 | SIMPLE      | employees | NULL       | const | PRIMARY       | PRIMARY | 4       | const |    1 |   100.00 | NULL  |
+----+-------------+-----------+------------+-------+---------------+---------+---------+-------+------+----------+-------+

eq_ref 执行连接查询时，如果被驱动表是通过主键或者是唯一索引匹配的，那么访问方法就是 eq_ref

ref 执行连接查询时，当通过普通的二级索引列与常量进行等值匹配的方式来查询某表时，那么访问方法就是 ref

ref_or_nll 当对普通的二级索引进行等值匹配且该索引列的值也可以是 NULL 时

fulltext 全文索引

index_merge 索引合并的方式对表进行查询

range 如果使用索引获取某些扫描区间的记录，那么就可能使用到 range 访问方法。

1
2

mysql> explain select * from s1 where key1 in ('a', 'b', 'c');
mysql> explain select * from s1 where key1 > 'a' and key1 < 'b';

index 当可以使用索引覆盖，但需要扫描全部的索引记录时（key_part3 和 key_part2 是联合索引）

对于使用 innodb 存储引擎的表

  • 二级索引叶子节点的记录只包含索引列和主键列的值
  • 聚簇索引叶子节点中包含用户定义的全部列以及一些隐藏列，所以扫描全部二级索引记录的代表比扫描全部聚簇索引记录的代价更低一些。

possible_keys & key & key_len

possible_keys 列表示某个查询语句中，对某个表执行单表查询时可能用到的索引由哪些。

key 列表示实际用到的索引有哪些。比如下面这个查询。

1
2
3
4
5
6

mysql> explain select * from dept_emp where emp_no = 10001 and dept_no = 'd005';
+----+-------------+----------+------------+-------+-----------------+---------+---------+-------------+------+----------+-------+
| id | select_type | table    | partitions | type  | possible_keys   | key     | key_len | ref         | rows | filtered | Extra |
+----+-------------+----------+------------+-------+-----------------+---------+---------+-------------+------+----------+-------+
|  1 | SIMPLE      | dept_emp | NULL       | const | PRIMARY,dept_no | PRIMARY | 8       | const,const |    1 |   100.00 | NULL  |
+----+-------------+----------+------------+-------+-----------------+---------+---------+-------------+------+----------+-------+

上述执行计划的 possible_keys 列有 PRIMARY 和 dept_no，表示该查询可能用到主键索引和二级索引。

然后 key 列的值是是 PRIMARY 主键索引，表示经过查询优化器计算不同索引的使用成本后，最后决定用主键索引来执行查询。

另外需要注意一点，possible_keys 列中的值并不是越多越好，可以使用的索引越多，查询优化器在计算查询成本时花费的时间就越长。

key_len 表示使用的索引长度，key_len 可以衡量索引的好坏，key_len 越小索引效果越好。

rows & filtered

在查询优化器决定是哟个全表扫描的方式对某个表执行查询时，执行计划的 rows 列就代表该表的估计行数。

如果使用索引来执行查询，执行计划的 rows 列就代表预计扫描的索引记录行数。比如下面这个查询：

mysql> explain select * from dept_emp where dept_no > 'd008';
+----+-------------+----------+------------+-------+---------------+---------+---------+------+-------+----------+-----------------------+
| id | select_type | table    | partitions | type  | possible_keys | key     | key_len | ref  | rows  | filtered | Extra                 |
+----+-------------+----------+------------+-------+---------------+---------+---------+------+-------+----------+-----------------------+
|  1 | SIMPLE      | dept_emp | NULL       | range | dept_no       | dept_no | 4       | NULL | 41384 |   100.00 | Using index condition |
+----+-------------+----------+------------+-------+---------------+---------+---------+------+-------+----------+-----------------------+

执行计划的 rows 列的值是 41384，这意味着查询优化器在分析完使用 dept_no 执行查询的成本之后，觉得满足 dept_no > ‘d008’ 条件的巨鹿只有 41384 条。

filtered

下面的执行计划，条件 dept_no > ‘d007’ 用来形成扫描区间，从 rows 列看可以满足的条件有 331570 条

而 filtered 列代表的是，查询优化器预测出这 331570 条记录中有多少条（331570 * 26.26%）记录满足其余的其余的搜索条件。

Using join buffer（Block Nested Loop） 在连接查询的执行过程中，当被驱动表不能有效地利用索引加快访问速度时，MySQL 一般会为其分配一块名为连接缓冲区（Join Buffer）的内存块来加快查询速度。

Using temporary 在许多查询的执行过程中，MySQL 可能会借助临时表来完成一些功能，比如去重、排序之类的。

比如我们在执行许多包含 DISTINCT、GROUP BY、UNION 等子句的查询过程中，如果不能有效利用索引来完成查询，MySQL 很有可能通过建立内部的临时表来执行查询。

如果查询中使用到了内部的临时表，在执行计划的 Exrta 列将会显示 Using temporary

另外注意，一般 group by 语句会自带 order by 语句的行为，如果我们不想为 group by 的查询排序，则需要显式地写上 order by null
