性能优化

数组传参的性能问题

在 Go 中，与 C 数组变量隐式作为指针使用不同，Go 数组是值类型，赋值和函数传参操作都会复制整个数组数据。

func main() {
    arrayA := [2]int{100, 200}
    var arrayB [2]int

    arrayB = arrayA

    fmt.Printf("arrayA : %p , %v\n", &arrayA, arrayA)
    fmt.Printf("arrayB : %p , %v\n", &arrayB, arrayB)

    testArray(arrayA)
}

func testArray(x [2]int) {
    fmt.Printf("func Array : %p , %v\n", &x, x)
}

打印结果：

1
2
3

arrayA : 0xc4200bebf0 , [100 200]
arrayB : 0xc4200bec00 , [100 200]
func Array : 0xc4200bec30 , [100 200]

可以看到，三个内存地址都不同，那这会导致什么问题呢？

假设数组大小有 100 万，在 64 位机器上就需要花费大约 800W 字节，即 8MB 内存。每次传参，都要消耗掉大量的内存。

所以需要传数组的情况下，一定要传地址，而不是传值。比如下面这样。

func main() {

    testArrayPoint1(&arrayA) // 1.传数组指针
    arrayB := arrayA[:]
    testArrayPoint2(&arrayB) // 2.传切片
    fmt.Printf("arrayA : %p , %v\n", &arrayA, arrayA)
}

func testArrayPoint1(x *[2]int) {
    fmt.Printf("func Array : %p , %v\n", x, *x)
    (*x)[1] += 100
}

func testArrayPoint2(x *[]int) {
    fmt.Printf("func Array : %p , %v\n", x, *x)
    (*x)[1] += 100
}

func Array : 0xc4200b0180 , [100 300]
arrayA : 0xc4200b0140 , [100 400]

现在就算是传入 10 亿的数组，也只需要在栈上分配一个8个字节的内存指针，存数组首地址就可以了。

但是切片传参就没这个问题，切片本身结构只占用 24 字节（64 位下），所以传值和传地址都无所谓的。

逃逸分析

逃逸简单来说就是 Golang 编译器经过分析决定变量分配到堆上而不是栈上。（通常变量能存储到栈上，优先栈上，减少 GC 对象的时间）

1

$ go build -gcflags '-m=1 -l'
  • -l 是禁止内联，-m 是打印优化决定，后面指定的数字越大越详细
  • 重点关注 escapes to heap 和 moved to heap 附近的打印

如果能不用 interface{} 代替泛型，就不用。因为对象会逃逸到堆。

子切片导致无法 GC 问题

在已有切片的基础上进行切片，不会创建新的底层数组。因为原来的底层数组没有发生变化，内存会一直占用，直到没有变量引用该数组。

推荐的做法是用 copy 替代 re-slice。

1
2
3
4
5
6
7
8
9

func lastNumsBySlice(origin []int) []int {
    return origin[len(origin)-2:]
}

func lastNumsByCopy(origin []int) []int {
    result := make([]int, 2)
    copy(result, origin[len(origin)-2:])
    return result
}

上面两个函数的作用是一样的，取 origin 切片的最后 2 个元素

  • 第一个函数直接在原切片基础上进行切片。
  • 第二个函数创建了一个新的切片，将 origin 的最后两个元素拷贝到新切片上，然后返回新切片

[]byte 和 字符串的高效互转

从底层结构上来看 string 可以看做 [2]uintptr，[]byte 切片类型可以看做 [3]uintptr，之间的关系就像下图。

如何 string 转 []byte 呢？

只需构建出 [3]uintptr{ptr, len, len}，生成的 cap 字段对读取操作没有影响，但如果要往转换后的 slice append 元素可能有问题，原因是这样做，slice 的 cap 属性是随机的，可能是大于 len 的值，那么 append 时就不会新开辟一段内存存放元素，而是在原数组后面追加，如果后面的内存不可写就会产生 panic。

而 []byte 转 string 更简单，直接转换指针类型即可，忽略 cap 字段。

实现如下

1
2
3
4
5
6
7
8
9

func stringTobyteSlice(s string) []byte {
    tmp1 := (*[2]uintptr)(unsafe.Pointer(&s))
    tmp2 := [3]uintptr{tmp1[0], tmp1[1], tmp1[1]}
    return *(*[]byte)(unsafe.Pointer(&tmp2)) 
}

func byteSliceToString(bytes []byte) string {
    return *(*string)(unsafe.Pointer(&bytes))
}

上面使用了 unsafe.Pointer 来转换不同类型的指针，没有发生底层数据的拷贝，因此效率和普通转换比起来较高。

高效拼接字符串

Go中可以使用“+”合并字符串，但是这种合并方式效率非常低，每合并一次，都是创建一个新的字符串，就必须遍历复制一次字符串。

使用 bytes.Buffer 来组装字符串，不需要每次创建合并字符串了，只需要将添加的字符串放在缓冲区末尾即可

package main

import (
    "bytes"
    "fmt"
)

func main() {
    var buffer bytes.Buffer
    for i := 0; i < 2000; i++ {
        buffer.WriteString("a")
    }
    fmt.Println(buffer.String())
}

// InterfaceStruct 定义了一个 interface{} 的内部结构
type InterfaceStruct struct {
    pt uintptr // 到值类型的指针
    pv uintptr // 到值内容的指针
}

// ToInterfaceStruct 将一个 interface{} 转换为 InterfaceStruct
func ToInterfaceStruct(i interface{}) InterfaceStruct {
    return *(*InterfaceStruct)(unsafe.Pointer(&i))
}

func IsNil(i interface{}) {
    fmt.Printf("i value is %+v\n", ToInterfaceStruct(i))
}

func main() {
    var sl []string
    IsNil(sl)
    IsNil(nil)
}

for range 对象被复制的陷阱

for range 的对象会被动的做值拷贝。举个例子

  • 如果是数组拷贝，本次循环的 k, v 迭代不会被改变，因为是从副本里读的
  • 而引用类型底层数据并没有拷贝，所以即使在循环里修改内容，也能正常的迭代出 k, v 的最新修改值

所以用 for range 遍历数组时，开销随着数组的大小直线上升（注意！）

    for i, v := range a { // i, v 从 a 对象的复制副本里提取
        if i == 0{

已关闭 channel 的一个陷阱

如果在 for select 语句中发现某个分支的通道已关闭，那么这个分支会一直被执行。

为了防止再次进入这个分支，可以把这个 channel 重新赋值为 nil，那么这个 case 就不会触发了。

for {
    select {
    case x, open := <-inCh1:
        if !open {
            inCh1 = nil
            break
        }
        out<-x
    case x, open := <-inCh2:
        if !open {
            inCh2 = nil
            break
        }
        out<-x
    }

    // 当ch1和ch2都关闭是才退出
    if inCh1 == nil && inCh2 == nil {
        break
    }
}

package main

    "fmt"
)

func main() {
    myNum := []int{10, 20, 30, 40, 50}
    for k, v := range myNum {
        fmt.Println(k, v)
    }

    newNum := myNum[1:3]
    newNum = append(newNum, 60)

    fmt.Println("after")
    for k, v := range myNum {
        fmt.Println(k, v)
    }
}

可以看到，对newNum调用append()函数却改变了myNum的数据。

注意 append 的一些事项：

  1. 当 append 之后的长度小于等于 cap，将会直接利用原底层数组剩余的空间。
  2. 当 append 后的长度大于 cap 时，则会分配一块更大的区域来容纳新的底层数组，并把原来的数据复制过去。

因此，为了避免内存发生拷贝，如果能够知道最终的切片的大小，预先设置 cap 的值能够获得最好的性能。

对数组进行切片的注意事项

下面的代码中，对数组切片后，cap 是数组的长度，len 是切片后的长度。ptr 指向数组的内存空间。

当我们对切片进行修改、append 或者 copy，都会影响到数组本身。编程时需要注意。

func main() {
	array := [4]int{10, 20, 30, 40}
	slice := array[0:2]
	newSlice := append(slice, 50)

	fmt.Printf("array addr = %p\n", &array)
	fmt.Printf("Before slice = %v, Pointer = %p, len = %d, cap = %d\n", slice, &slice[0], len(slice), cap(slice))
	fmt.Printf("Before newSlice = %v, Pointer = %p, len = %d, cap = %d\n", newSlice, &newSlice, len(newSlice), cap(newSlice))
	newSlice[1] += 10
	fmt.Printf("After slice = %v, Pointer = %p, len = %d, cap = %d\n", slice, &slice[0], len(slice), cap(slice))
	fmt.Printf("After newSlice = %v, Pointer = %p, len = %d, cap = %d\n", newSlice, &newSlice, len(newSlice), cap(newSlice))
	fmt.Printf("After array = %v\n", array)
}

map 遍历是无序的

hash 表在扩容缩容时会发生 rehash，顺序会发生变化，但是没有扩缩容的时候，顺序是有保证的。所以 golang 为了让程序员不依赖这种不可靠的保证，就干脆遍历的时候加入随机数，然后不管什么时候遍历，顺序都是不保证的。

所以要想有序输出 map，则要先按 key 排序需要取出 key，对 key 排序，再遍历一次输出 value。

package main

import (
    "fmt"
    "sort"
)

func main() {
    // To create a map as input
    m := make(map[int]string)
    m[1] = "a"
    m[2] = "c"
    m[0] = "b"

    for k, v := range m {
        fmt.Println("Key:", k, "Value:", v)
    }


    // To store the keys in slice in sorted order
    var keys []int
    for k := range m {
        keys = append(keys, k)
    }
    sort.Ints(keys)

    fmt.Println("after")
    // To perform the opertion you want
    for _, k := range keys {
        fmt.Println("Key:", k, "Value:", m[k])
    }
}
